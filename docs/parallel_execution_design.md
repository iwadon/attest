# attest 並列実行設計

この文書は attest フレームワークにおけるテスト並列実行機能の設計・実現可能性を検討するものです。

---

## 1. 概要

テストの実行時間を短縮するため、複数のテストを並列に実行する機能を提供します。

### 目標
- テスト実行時間の短縮
- マルチコア環境の効率的な利用
- 既存のテストコードとの互換性維持

---

## 2. 基本仕様（案）

### 2.1 コマンドラインオプション
```
--jobs=N
```
- `N`: 最大同時実行数（1以上の整数）
- デフォルト: 1（並列実行なし）
- `N=0` または `--jobs=auto`: CPU コア数を自動検出

### 2.2 実行単位
- テスト関数単位で並列化
- Fixture の setup/teardown は各テストごとに独立して実行

---

## 3. 技術的課題

### 3.1 スレッド安全性
- **グローバル変数**: 現在の実装でテスト登録・結果収集に使用
  - レジストリ: 読み取り専用なので問題なし
  - 結果収集: mutex で保護が必要

- **標準出力**: 複数スレッドからの出力が混在する可能性
  - バッファリング機構が必要
  - テストごとに出力をキャプチャし、完了時に順次出力

### 3.2 実行順序の保証
- 出力は登録順を維持
- 実行順序自体は不定（並列実行のため）
- 依存関係のあるテストは順次実行する必要あり（スコープ外）

### 3.3 共有リソース
- ファイルシステム、環境変数、グローバル状態など
- ユーザー責任で管理（ドキュメントで注意喚起）
- 将来的に `ATT_SERIAL` マクロで明示的に順次実行を指定可能にする案

### 3.4 クロスプラットフォーム対応
- POSIX: pthreads
- Windows: Win32 threads
- 標準 C11: threads.h（オプション機能）

---

## 4. 実装アプローチ

### 4.1 スレッドプール方式
```
[Main Thread]
  ↓
[Worker Thread 1] [Worker Thread 2] ... [Worker Thread N]
  ↓                ↓                     ↓
[Test Queue] → テストを順次取得して実行
  ↓
[Result Buffer] → 完了したテストの結果を登録順にマージ
```

### 4.2 データ構造
```c
typedef struct {
    void (*test_fn)(void);
    const char *suite;
    const char *name;
    // fixture info...
} test_task;

typedef struct {
    test_task *tasks;
    size_t count;
    size_t next;
    pthread_mutex_t lock;
} test_queue;

typedef struct {
    char *output;
    int status; // pass/fail/skip
    double duration;
} test_result;

typedef struct {
    test_result *results;
    size_t count;
    pthread_mutex_t lock;
} result_buffer;
```

### 4.3 実行フロー
1. すべてのテストをキューに登録
2. N 個のワーカースレッドを起動
3. 各ワーカーはキューからテストを取得して実行
4. 結果をバッファに格納（mutex 保護）
5. すべてのワーカーが完了後、結果を登録順に出力

## 6. パフォーマンス見積もり

### 前提
- テスト数: 1000
- 平均実行時間: 10ms/test
- CPU コア数: 8

### 理論値
- 順次実行: 10s
- 並列実行（8 並列）: 10s / 8 = 1.25s
- オーバーヘッド込み: 約 1.5-2s

### 制約
- I/O バウンドなテストでは効果が薄い
- セットアップコストが大きい場合は逆効果の可能性

---

## 7. リスクと対策

| リスク | 対策 |
|--------|------|
| デッドロック | タイムアウト機構と併用 |
| 出力の混在 | バッファリングで順序保証 |
| 共有リソース競合 | ドキュメントで注意喚起 + 将来的に ATT_SERIAL サポート |
| デバッグ困難化 | 各テストの出力を明確に分離 |

---

## 8. 互換性

- デフォルト動作（`--jobs=1`）では既存の挙動を維持
- 並列実行は opt-in
- 既存のテストコードに変更不要
